Part 1: Critical Security & Database Architecture Vulnerabilities
1. Data Leakage of Hidden Roles (The Spectator/Graveyard Security Risk)
The Issue:The game design states that dead players (Spectators/Graveyard) and the Game Master (GM) can see all players' real roles, while active players must only see their own. In a client-side Firebase structure where players listen to/rooms/{roomId}/players, the entire player array is downloaded to the client. A player can open the browser console, inspect the local database state, and instantly identify all Werewolves, Neutrals, and Villagers.
Architectural Flaw:Lack of read-level isolation in the database schema. Client-side hiding of UI elements using CSS classes or conditional rendering provides no real data security.
Mitigation Strategy:
ImplementDatabase Path Isolation. Split the room's data structure into three distinct segments:
/rooms/{roomId}/public: General states (alive/dead status, public votes, phase, layout themes) readable by everyone.
/rooms/{roomId}/private/{playerId}: Contains specific sensitive data (role, mailbox, target selections) read-restricted via Firebase Security Rules toonlythat specific authenticatedplayerId.
/rooms/{roomId}/secure_roles: Contains the master list of all roles. Only the Host (GM) can read this path. Spectators can gain read access to this path only when their state is updated todeadvia a server-enforced trigger.
Alternatively, useFirebase Cloud Functionsto calculate game states and strip out role information from the payloads returned to living players, ensuring clients never receive unauthorized data.
2. Network Overhead of Real-Time Speech Synchronization
The Issue:During the Defense Speech Phase (Step 2), the accused player's keystrokes are synced to Firebase with a target latency of under 100ms. If this typing state is written directly into the main room state or player object, it triggers database triggers and forces every listener to re-evaluate the entire room's payload. This causes significant bandwidth consumption, UI stuttering, and potential rate-limiting.
Architectural Flaw:Mixing heavy, low-frequency structural data with light, high-frequency ephemeral data.
Mitigation Strategy:
Create anEphemeral / Real-time Messaging Channel(e.g.,/rooms/{roomId}/ephemeral/defenseSpeech) that handles only the raw text stream.
Living players subscribe only to this leaf node during Step 2.
Throttle client-side write frequency using a debounce function (eg, every 150–200ms instead of on every keydown event) to maintain a smooth user experience while reducing database load.
3. Client-Side Game Resolution Dependency (Host-Crash Vulnerability)
The Issue:If state transitions (such as determining who died during the night, calculating Witch saves, or managing countdown timers) are calculated purely on the Host's client and then written back to Firebase, the game becomes highly vulnerable. If the Host closes their tab, experiences a network drop, or lags, the game state freezes, conflicts arise, or the room becomes corrupted.
Architectural Flaw:Peer-to-peer state authority delegated to a single volatile client.
Mitigation Strategy:
Implement aServerless State Machineor a decentralized failover mechanism.
If relying strictly on Firebase client-side code: write aHost Heartbeat Monitor(/rooms/{roomId}/host/heartbeat). If the active Host is unresponsive for more than 10 seconds, the client with the longest connection duration (next in line) is promoted to Host, claims the write lock, and continues executing state transitions.
Ideally, useFirebase Cloud Functionsdriven by Firestore/RTDB triggers or scheduled tasks to handle state progression (eg, phase transitions on timer exhaustion) independently of any client.

2. Resolving Conflicting Night Actions (The Priority Tick Engine)
The Issue:In a single night, a player might be:
Protected by a Guard.
Bitten by Werewolves.
Poisoned by the Witch.
Saved by the Witch.
Silenced by the Silencer Wolf.
Targeted by the Avenger's sleep potion.
Swapped by the Phantom Wolf.
Converted by a Missionary.
Mitigation Strategy:
Implement aPriority Tick Enginethat resolves night actions in a strict, sequential order. Actions are not resolved in the order they are submitted; instead, they are held in a buffer and resolved at dawn using this priority order:
Tick	Action Phase	Roles Involved	Logical Outcome
1	Illusion & Traps	Phantom Wolf, Eradicator	Swaps identity flags or sets triggers before actions occur.
2	Redirection	Manipulator	Re-routes targeting pointers in the action buffer.
3	Protection & Reflection	Guard, Reflector, Angel	Resolves defensive fields, clears debuffs, and processes reflection targets.
4	Block / Silence	Silencer, Avenger (Gây Mê), Cat	Disables target's active abilities for the remaining tick evaluation.
5	Alignment Shift	Cupid, Missionary, Prime	Binds lovers, converts targets (important for checking victory conditions).
6	Lethal Actions	Wolf Pack, Serial Killer, Witch, Arsonist	Apply damage or death flags. Evaluates Guard or Witch potion interactions.
7	Retaliation	Hunter, Carver	Triggers death-throes behavior (eg, Hunter shot) if a player dies in Tick 6.
3. State Conflict on Converted Teams (Neutral Alignments)
The Issue:A player is targeted by the Missionary (isConverted: true) and the Vampire (isVampireVictim: true) on the same night. Who gets ownership of the player's soul? If they are converted, does the Vampire's bite still turn them into a Vampire victim, or does their holy status protect them?
Mitigation Strategy:
Establish aMutual Exclusion Rulein the Tick Engine.
Define team priority hierarchies: for example,Holy Conversion (Missionary) takes precedence over Undead Infection (Vampire), or vice versa. When one status is written to the player's private state, it clears the opposing flag and registers a failure log for the secondary action, updating the respective players' Mailboxes accordingly.
4. Disconnected Active Roles Block the Game
The Issue:If a key role (eg, Seer or Witch) disconnects during their night action turn, the night phase may stall indefinitely while waiting for their choice.
Mitigation Strategy:
Introduce anAuto-Pass / AI Fallback Engine.
If a player has been offline (online: false) or idle (no heartbeat) for more than 15 seconds into their active phase, the server automatically executes a "Pass" action (or triggers a random/basic auto-select action for Werewolves) and advances the phase.
Part 3: UI/UX Synchronization & Mobile Responsiveness Flaws
1. Multiple Stacked Modals & View Collisions
The Issue: index.htmldefines several independent modal overlays (#modal-alert,#modal-alert-action,#flashcard-modal,#mailbox-parchment-modal,#modalvote,#target-modal,#victory-screen-modal). If a player has their Mailbox parchment open when the day phase abruptly transitions to a voting phase, or the game ends, multiple modals will stack on top of each other. This locks the interface, hides essential controls, or disrupts the user experience.
Mitigation Strategy:
Implement a centralModal Manager Classin your front-end architecture.
Modals should not be displayed by directly toggling class names in separate scripts. Instead, they should be queued and rendered based on priority:
Priority 1:Victory Screen / Game Over
Priority 2:Voting & Action Modals (Interactive phase requirements)
Priority 3:Information Popups (Mailbox, Flashcard)
When a higher-priority modal is triggered, the manager automatically closes or minimizes lower-priority modals, saving their state for later retrieval.
2. Layout Elements with Duplicate IDs
The Issue:In the provided HTML structure, there are layout panels with duplicated structural components (eg,#col-leftcontaining both the GM timeline and the player mailbox, and right-column tabs managing complex sub-panels). In standard JavaScript,document.getElementById()only returns the first matching element. If IDs are duplicated across responsive breakpoints or sub-panels, script selectors will fail to bind on mobile layouts, causing broken buttons and unresponsive tabs.
Mitigation Strategy:
EnforceUniqueness ID. Change layout-specific wrappers to use distinct IDs (eg,#col-left-gm,#col-left-player,#col-right-chat,#col-right-config).
WornCSS Class Selectorsor data-attributes (data-role="timeline") for responsive visual hooks, ensuring scripts can target elements reliably regardless of screen size.

1. Database Partitioning (Firebase Realtime Database)
Separate public and private states to secure the game. Living players must only have read access to/rooms/{roomId}/publicand/rooms/{roomId}/private/{auth.uid}. This prevents players from reading the roles of others from the client console.
2. The Deterministic Resolution Loop
Intick-engine.js, hold all submitted night actions in/rooms/{roomId}/submitted_actionsuntil the night timer expires or all active players have submitted their choices. Then, resolve the actions sequentially using the priority levels fromPart 2, Section 2, and write the final output to the players' status fields and mailboxes.
3. Ephemeral Communication for Speeches
Route the live defense speech stream to an ephemeral path, such as/rooms/{roomId}/ephemeral_defense. This prevents updates from triggering listeners on the main room state, reducing database load and ensuring a smooth typing experience on mobile.
4. The Modal Queue System
Inui-manager.js, manage overlay screens using a priority queue. This ensures that critical game updates (like the Gavel Strike animation or the Victory screen) automatically minimize or defer less critical windows (such as the Mailbox or settings panel), preventing layout locks.
